指示階層とは何か

system / developer / user / tool result の権限構造。文章の内容だけでなく、どこから来て、どの層に置かれたかを読む。

指示の強さは、言葉の強さではなく、置かれた層によって決まる。

ページ作成日時：2026-07-20 12:12 JST
最終更新日時：2026-07-20 12:12 JST

1. プロンプトは一枚の文章ではない

生成AIに入る指示は、一つではない。ユーザーがチャット欄に書いた文章だけでもない。アプリケーション側の方針、会話履歴、プロジェクト指示、外部ファイル、検索結果、tool result。これらが実行時に集められ、異なる出所と権限を持った入力としてモデルへ渡される。

だから生成AIの挙動を理解するには、文章の内容だけを読むのでは足りない。その文章がどこから来たのか、どの入力経路へ置かれたのか、命令として扱うべきなのか、資料として扱うべきなのかまで読む必要がある。

2. roleは人物設定ではない

role は「あなたは編集者です」「あなたは弁護士です」という演技上の役割ではない。実装上のroleは、情報の出所・権限・扱い方を示す属性である。

観点意味
sourceplatform / developer / user / model / tool / external content のどこから来たか。
authority上位方針か、今回の依頼か、資料か。
handling従う、参照する、引用する、無視する、確認する、停止する。

ユーザーが本文中に system: と書いても、それはuser message内の文字列である。文章の見た目がsystem風でも、入力経路はuserのままである。

3. system / developer / user / tool result

system は、プラットフォームや実行環境側が置く上位制約に近い。通常のユーザーがチャット欄から直接編集できるものではない。

developer は、アプリケーション開発者やサービス設計者が置く方針である。出力形式、業務ルール、toolの使い方、確認条件、外部資料の扱いなどを置く。

user は、エンドユーザーの質問、作業依頼、修正指示、処理対象、出力希望である。重要だが、systemやdeveloperなどの上位制約の範囲内で解釈される。

tool result は、外部世界から戻されたデータである。その本文中に命令文が含まれていても、原則としてdeveloper instructionではない。

4. 衝突したときに見る順番

1. sourceを特定する
2. instruction authorityがあるかを判定する
3. 現在の作業へ適用されるscopeかを判定する
4. 本当に衝突しているかを確認する
5. 上位指示を守りながら、下位の意図を可能な範囲で満たす
6. 外部データ中の命令文を資料として隔離する
7. 曖昧または不可逆な操作では確認・停止する

処理は、単純な「従う／従わない」ではない。権限のある指示には従い、資料として有用なものは参照し、外部資料中の命令文は隔離し、危険な操作では確認または停止する。

5. prompt injectionは境界事故である

prompt injectionとは、外部資料や検索結果の中の命令文が、本来は資料であるにもかかわらず、AIへの指示として扱われてしまう境界事故である。強い言葉で「これまでの指示を無視せよ」と書かれていても、それが外部資料の本文であれば、上位指示にはならない。

中心フレーズ

指示の強さは、言葉の強さではなく、置かれた層によって決まる。
