生成AIのしくみ 超詳解 07

停止条件と検証ループ

agentはいつ続行し、いつ止まるのか。agentic systemを、止まれる制御ループとして読む。

agentic loopとは、モデルがただ動き続けることではなく、続行または停止を選ぶ制御ループである。

ページ作成日時：2026-07-20 12:12 JST
最終更新日時：2026-07-20 12:12 JST

1. agentic loopは無限ループではない

agentという言葉には、「自律的に動き続けるもの」という印象がある。ユーザーが目的を与え、AIが計画を立て、toolを呼び、結果を見て、また考え、また実行する。

しかし、信頼できるagentic systemを設計するうえで重要なのは、どう続けるかだけではない。いつ止まるかである。

receive task
  ↓
observe context
  ↓
decide next action
  ↓
act / call tool / answer / ask user / handoff
  ↓
observe result
  ↓
verify
  ↓
continue or stop

2. stopはfailureではない

停止は必ずしも失敗ではない。正しく止まることはagentの能力である。

状態意味
completed成功条件を満たして完了した。
blocked権限、ファイル、接続、情報などの条件が足りない。
needs_userユーザー判断が必要。
manual_review人間の確認に戻すべき状態。
handoff別の担当・別のworkflowへ渡す状態。
stop ≠ failure
blocked ≠ failed
needs_user ≠ failed
handoff ≠ failed

3. success criteriaがないagentは止まれない

agentが止まれない最大の理由は、成功条件が曖昧なことである。「この件を調べて、いい感じにまとめて」だけでは、何件調べればよいのか、どのsourceを信頼するのか、どの深さまで読むのか、どこで止まるのかが分かりにくい。

stop when:
  - required sources have been checked
  - contradictions have been noted
  - output format is complete
  - no unresolved required fields remain
  - budget is not exceeded
  - no approval-gated action remains pending

4. loop stateとbudget

agentic loopを制御するには、現在の状態を持たなければならない。何step目か、どのtoolを何回呼んだか、どの試行が失敗したか、あと何回retryできるかを記録する。

budgetには、turn、tool call、time、cost、token、retry、risk、side effectがある。特にread-onlyなtool callとstate-changingなtool callは分ける必要がある。

5. verification loop

agentが「完了しました」と言うことと、実際に完了していることは違う。modelの自己申告、tool resultの成功、valid JSONの返却だけではtask完了とは限らない。

model says completed ≠ task completed
tool call succeeded ≠ task goal achieved
valid JSON ≠ correct answer
file created ≠ useful artifact

verification loopとは、成果物の存在、差分、test、外部データ照合、引用、承認記録などを通じて、taskが本当に成功条件を満たしたかを確認する仕組みである。

中心フレーズ

agentic loopとは、自律的に動き続ける仕組みではなく、続行・検証・停止・handoffを選ぶ制御ループである。
