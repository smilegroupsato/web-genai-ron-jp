生成AIのしくみ 超詳解 01

プロンプトは実行時にどうコンパイルされるのか

ユーザーの一文は、そのままモデルへ流れ込むのではない。指示、文脈、ツール、制約が組み合わされて入力環境が作られる。

プロンプトとは、ユーザーが書いた文章そのものではなく、実行時に組み立てられた入力環境である。

ページ作成日時：2026-07-20 12:12 JST
最終更新日時：2026-07-20 12:12 JST

注記：このページの説明は、公開仕様、一般的なLLM実装、API設計上の概念、および理解のためのモデルを組み合わせたものです。特定サービスの非公開内部実装を、そのまま断定するものではありません。

1. ユーザーの文章だけがプロンプトではない

ユーザーの画面には、一つの入力欄がある。そこに文章を書き、送信する。だから、表面上は「チャット欄に書いた文章」がプロンプトに見える。

しかし、実行構造として見ると、ユーザーの文章は単独でモデルに渡されるとは限らない。その背後には、システム側の指示、アプリケーション側の指示、プロジェクト指示、カスタム指示、会話履歴、保存メモリ、添付ファイル、検索結果、ツール定義、現在利用できる機能、出力形式の制約などがある。

ユーザーが見ているもの
  ↓
チャット欄の文章
  ↓
実行環境
  ├─ 指示 system / developer / project
  ├─ 文脈 history / memory / files / search
  └─ 行動空間 tools / tool_choice
  ↓
モデルに渡される入力構造

同じ「この文章を要約してください」という依頼でも、直前の会話履歴が入っている場合、プロジェクト指示が入っている場合、添付ファイルの内容が入っている場合、Web検索結果が入っている場合、利用可能なツールが違う場合で、モデルが受け取る入力環境は変わる。

2. instructions / input / tools は分けて考える

自然な日本語では、一つの依頼文の中に、役割指定、処理対象、操作希望が混ざっている。

たとえば「あなたは編集者です。以下の文章をWeb掲載用に整えて、GitHubに書いてください。」という一文は、人間には自然に読める。しかし実行構造としては、少なくとも三つに分けて考える必要がある。

部分実装上の読み方
あなたは編集者ですinstructionsに近い。モデルの振る舞いを決める指示。
文章をWeb掲載用に整えるinputに対するタスク指示。
GitHubに書くtoolsと実行権限が関わる外部操作。

「文章を整える」だけならモデルの出力で完結する。しかし「GitHubに書く」は、外部システムへの書き込みである。どのリポジトリのどのファイルに書くのか、既存内容を読むのか、差分確認を行うのか、失敗時にどこで止まるのかも関係する。

3. UI上の擬似階層と実行環境上の実階層

ユーザーは本文中に system: や developer: と書くことができる。しかし、それは実行環境上の本物のsystem messageやdeveloper messageになったわけではない。見た目のラベルと、入力経路上のroleは違う。

強い言葉で書かれたuser messageが、上位指示へ昇格するわけではない。逆に短い一文でも、実行環境がdeveloper instructionとして置いたものは、user messageより強い制約として扱われることがある。

4. prompt compilationという理解モデル

ここでいう prompt compilation は、非公開内部実装の固有名ではない。ユーザー入力、上位指示、履歴、memory、検索結果、ファイル、tool定義、出力制約などが、実行時に入力構造として組み立てられる過程を理解するためのモデルである。

user message
  + instructions
  + history
  + memory
  + files / retrieval
  + tool definitions
  + output constraints
  = runtime input environment

5. 中心フレーズ

プロンプトとは、ユーザーが書いた文章そのものではなく、実行時に組み立てられた入力環境である。
